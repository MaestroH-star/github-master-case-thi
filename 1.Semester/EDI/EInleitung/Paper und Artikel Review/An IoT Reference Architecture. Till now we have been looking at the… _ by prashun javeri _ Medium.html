An IoT Reference Architecture

prashun javeri
4 min read
·
Jun 26, 2018

--

Till now we have been looking at the domain model specification for IoT with an emphasis on data architecture .We identified what kind of data does IoT require ,where we should be storing this data and how we should be storing it from here on we will look more at over all system architecture of the IoT platform .

ok so first lets look to breakdown the system into smaller components .The best place to start will be a reference architecture ok so the main components of the reference architecture are

  • Device Manager : Distributed devices can be controlled either directly or indirectly via a centralized agent , functionality can include over the air distribution of applications, data and configuration settings for all types of devices. Device management is intended to support the following uses: • Provisioning — Configuration of the device (including first time use), enabling and disabling features
  • Device Configuration — Allow changes to settings and parameters of the device
  • Software Upgrades — Provide for new software and/or bug fixes to be loaded on the device, including applications and system software
  • Fault Management — Report errors from the device, query about status of device
  • Identity and access management — Security will be one of the most important factors to consider when working with IoT so any IoT platform should have builtin security and access controls at each level
  • Identity Manager — Securing an open platform requires a robust security architecture and rigorous security programs.so any IoT platform should have a well defined security policy which drives access management. Identity Management is thus intended to support the following use cases
  • Provisioning and de-provisioning of accounts — allowing access to or revoking access to the platform as a whole.
  • Single Sign on — Single Sign on for all devices and applications based on pre agreed upon access rights • Role based Access Control for Applications : Restricting access to certain applications even if the device is accessible and user has signed in using Single Sign on .
  • Attribute based Access Control for Devices — Restricting access to certain devices even if the application is accessible and user has signed in using Single Sign on .
  • Enterprise Service Bus — The ESB is used to provide support for message oriented architecture. The ability to transform messages has the benefit of allowing applications to be much more decoupled from each other. it also has the added benefit of providing device to device communication when the two devices communicating are using different network layer protocols.

The ESB is intended to support the following uses:

  • Realtime Data Transformation,Enrichment and cleansing required to integrate with external systems
  • Monitor and control routing of message exchange between External Enterprise systems and IoT system

Event Bus :This provides support for handling events .An event-driven architecture consists of event creators and event consumers. The creator, which is the source of the event, only knows that the event has occurred. Consumers are entities that need to know the event has occurred; they may be involved in processing the event or they may simply be affected by the event. The Event bus is intended to support the following uses:

  • Node to node communication (between fog Nodes) : often the devices communicating with each other will not be part of the same cluster and an event generated by one device or system may want a device or system on another cluster to react to it .this kind of communication can take time in a message oriented architecture if the message queue is backed up
  • Provides for system and device decuopling

Aggregation Engine : the distributed nature of the IoT platform required that data be aggregated from multiple nodes into a central repository so that it can be analyzed

  • The Aggregation is intended to support the following uses: • Aggregate Log files from fog nodes and devices
  • Aggregate Events from fog nodes
  • Get aggregates on KPI from fog nodes
  • Aggregate system performance data from fog nodes and devices

Rules Engine : is used to define the rules which drive the business process and thus define the conditions under which the participating systems will generate events that are required for either for the completion of the business process or for some corrective action to be taken

The Rules Engine is intended to support the following uses:

  • Sensor To Device Interactions
  • Device to Device Interaction
  • User to Device Interactions
  • Workflow management
  • Alert management

Analytics engine : once the data had been gathered and aggregated it is often necessary to analyze this data against KPIs in order to green useful information out of this data . The Analytics Engine is intended to support the following uses:

  • Data grouping and clustering
  • Data ranking
  • Trend Analysis
  • Predictive Analysis
